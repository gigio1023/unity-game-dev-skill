# Game-Design Review Brief

Use only when a separate design pass would materially improve a feature.

Review the requested player experience, current mechanics, constraints, and
implementation evidence. Return:

1. intended player decision or feeling;
2. core loop and state transitions;
3. input-to-feedback contract;
4. failure, recovery, accessibility, and edge cases;
5. the smallest acceptance scenarios;
6. design risks that require product judgment.

Do not edit implementation files unless the user separately authorized a
change. Avoid expanding a bounded feature into a full design document.
